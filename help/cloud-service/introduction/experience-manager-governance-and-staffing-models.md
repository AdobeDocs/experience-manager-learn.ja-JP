# Adobe Experience Manager(AEM) — ガバナンスとスタッフのモデルとアーキタイプ

Adobeは、顧客体験リーダーとして、お客様が適切な人材とガバナンスのフレームワークを確保して運用効率を高めるために、どのように困難を伴うかを理解しています。 Adobeの業界で実証済みのガバナンスとスタッフのモデルを使用すると、堅牢なコンテンツとアセット管理の基盤を構築するためのツールと知識を習得できます。 この記事では、Adobe Experience Manager(AEM) プラットフォームを運用し、取り組みを最大限に活用する方法について説明します。

## 優れた運用フレームワークを作成

AEMを実行および操作できるように、次の要素を考慮します。

* 戦略的マイルストーンの実行 — 多くの戦略的マイルストーン、（パーソナライゼーション、マルチチャネル統合など） 適切なスタッフモデルが配置されていない場合は、実行できません。
* デジタル変換の基盤を作成する — 組織の最新化プロセスの最初のステップとして、AEMがよく使用されます。 基盤を設定すると、AEMを最大限に活用できます。
* ユーザーエンゲージメント — 戦術的な作業（更新ワークフロー、権限、CSS など）を実行するチームを置く ユーザーの欲しいものと与えられたものとの間にギャップがあればあるほど、ユーザーが不満を抱く可能性が高まります。 ユーザーがシステムに投資し、ソリューションに投資し、適切なオペレーティングモデルを用意しておくことが重要です。

それでは、正しいモデルは何ですか？ 作成する役割の適切なマトリックスは何ですか？

組織が大きく異なるのと同様に、AEMの設定も大きく異なるので、それぞれが異なる役割を担う必要が生じるので、具体的な回答はありません。 各業種、各業界、各チーム構造には異なる実装が必要です。 ただし、アーキタイプを確立することで、ベースラインを作成できます。

## アーキタイプ

アーキタイプは、特定の属性にマッピングされる、具体的で高レベルな役割のアイデアです。 次に、これを使用して、本当に必要なモデルを伝えるのに役立つ、基本的な前提を作成できます。 アーキタイプは、アーキタイプごとに 1 人に制限されないことに注意する必要があります。 例えば、DAM ライブラリアンは、技術的な経験を持つことができます。

には 2 つのオペレーションストリームがあります。 [!DNL AEM Sites] および [!DNL AEM Assets]:

1. 基本的な実行および操作の役割

   * テクニカルリソース — AEMエクスペリエンスを持つユーザーが権限を理解し、メタデータスキーマを更新できる
   * リリースマネージャー
   * 製品所有者 — これは、ソリューションに合わせて配置される役割です。 一部の製品所有者は、Analytics に関与できます。
   * DAM ライブラリアン — 統合的なフレームワークプロセスを羊飼いになれる人です。 このクリエイティブロールは他のロールと重複する場合があります。 ( 注意：これは、過去 5 年間に人気が高まった役割です。
   * クリエイティブ

1. 戦略と変換

   * 開発チーム — このチームは、重要な戦略的マイルストーンに取り組む際に必要です。
   * ビジネスアーキテクト：技術的なマイルストーンと戦略的な取り組みに役立つ要件を開発し、製品所有者を追加して補うことができます。
   * テクニカルアーキテクト — 企業レベルの理解を持ち、組織全体で常に存在する人物。 この役割は、DAM の真実の中心点として機能します。

### 高レベル [!DNL AEM Sites] 役割

1. 基本的な実行と操作

   **一般的なギャップ：** CSS 開発者は、コンポーネントの新しいスキンを作成します。 AdobeSr ビジネスコンサルタントの Joseph Van Buskirk 氏は、「コーク解除されたコンポーネントとスタイルシステムを取得する」ことをお勧めします。 これは、コスト削減を推進する役割です。 作成されるエクスペリエンスの 80%は、以前に作成したまたはコアコンポーネントを使用しておこなう必要があります。」 目標は、 CSS 開発者（またはフロントエンド開発チーム）を使用して、コアコンポーネントやカスタムコンポーネントを新しいスタイルで再利用することです。

   役割の例：

   * CSS 開発 — 新しいスタイルでコンポーネントを転用して、エクスペリエンスアーティファクトを作成します。
   * バックエンド開発 — 新しいコンポーネントを作成するか、コアコンポーネントを拡張できます。 正しく行われた場合、大きなアニメーションタスクが必要でない限り、この役割に複数の人を含めることはできません。
   * リリース管理 — コードのデプロイメントを監視し、現在のAdobeアカウントチームとして機能します。
   * 製品所有者 — BU との提携により、技術と戦略に関するビジョンを結び付けるは、メンテナンスタスクおよび機能強化を作成し、ソリューションのビジネスオーナーとしての役割を果たします。
   * 管理者作成者 — CSS スキンを更新し、コンテンツを更新および適用する作成者に対するガイダンスを提供します。 この役割は、ワークフロー設定に対して機能し、コンテンツ作成者が適用するガイダンスドキュメントを作成します。 注意：リリース 6.5 では、Adobeは、編集可能なテンプレートの使用をお勧めします。
   * コンテンツ作成者 — CSM で発生したコンテンツ、階層型所有権を適用し、通信上の問題や懸念事項を提供します。

1. 戦略と変換

   役割の例：

   * 開発チーム：AEMの知識を提供し、テクニカルアーキテクトと共に新しい変革的なマイルストーンを実行します。
   * テクニカルアーキテクト — 統合に関する知識を提供し、製品オーナーと連携して技術的なマイルストーンをマッピングし、AEMの深い技術的知識を提供します。
   * ビジネスアーキテクト — ユーザーストーリー用のタスクを作成し、製品オーナーが技術的およびビジネスマイルストーンの管理に役立ちます。

### 重要な留意点

**アーキタイプについて**  — 処理を遅くして、アーキタイプを理解し、分析します。 従うべき正しいモデルが 1 つもないことに注意し、クリエイティブで柔軟性を備えてください。

**ロードマップの理解**  — 一部の組織では、実行するマイルストーンが多数あります。 予想以上の技術リソースを割り当てる準備をしてください。

**内部リソースの活用**  — ギャップが予期せず生じる場合があります。 社外で検索するのではなく、内部チームメンバーを調達することで、より迅速にメンバーを入力できる場合があります。
